# ESIP-0001：核心消息模型

状态：Experimental

协议版本：0.1

内容类型：`application/json`

## 1. 目标与非目标

ESIP 为不同游戏、网页、侧车服务和自动化代理提供统一的语义边界。它规范消息结构、能力协商、路由上下文、版本、幂等和授权要求。

ESIP 0.1 不定义实时物理同步、全局数据库、统一账户系统、传输加密实现、消息代理部署或跨平台经济结算。适配器不能因实现 ESIP 就获得读取游戏全部状态的权限。

文中的“必须”“禁止”“应该”和“可以”表示不同强度的互操作要求。

## 2. 消息外壳

ESIP 消息使用 CloudEvents 1.0 风格的 JSON 属性。以下字段必须存在：

- `specversion`：固定为 `1.0`。
- `esipversion`：当前固定为 `0.1`。
- `id`：在同一 `source` 下唯一的消息 ID，也是重试幂等键。
- `source`：适配器的绝对 URI，例如 `esip://luanti/world-alpha`。
- `type`：带主版本的消息类型，例如 `io.evolution.action.requested.v1`。
- `kind`：`command`、`event`、`query` 或 `result`。
- `time`：消息产生时间，不作为游戏规则的唯一权威时钟。
- `datacontenttype`：固定为 `application/json`。
- `dataschema`：描述 `data` 的 schema 地址。
- `sequence`：同一 `source` 单调递增的非负整数。
- `data`：符合消息 schema 的对象。

`command` 和 `query` 必须指定 `target`。`correlationid` 关联一次跨平台流程，`causationid` 指向直接引发当前消息的消息。游戏 tick 可写入 `tick`，有时效性的请求应设置 `expiresat`。

## 3. 四种语义

- `command` 请求权威端改变状态。接收方可以拒绝，发送方不能把“已发送”视为“已执行”。
- `event` 描述已经发生且不可修改的事实。事件不能被消费端回写成新的历史。
- `query` 请求读取一组状态。查询也必须授权，因为状态可能包含敏感信息。
- `result` 是命令或查询的定向结果，包括状态快照与错误。

消息类型和 `kind` 的对应关系固定在 `src/interop/message-types.mjs` 与 AsyncAPI 文件中。

## 4. 权威与一致性

每一份可变游戏状态必须只有一个权威平台。例如 Luanti 世界对该世界中的玩家演化状态拥有最终写入权；网页只能发命令和显示已确认结果。

`action.requested` 命令必须携带 `expectedRevision` 实现乐观并发。权威端发现版本不一致时必须拒绝并返回 `revision_conflict`，不能静默覆盖较新的状态。

世界时间线注册表与单个 actor 状态是两份权威状态，必须使用独立 revision：

- `timeline.create.requested.v1` 必须携带当前 actor 的 `expectedStateRevision` 和世界的 `expectedRegistryRevision`。两者都匹配时，权威端登记新时间线、把创建者切换到该时间线，并返回 `timeline.created.v2`；任何冲突都不得产生部分写入。
- `timeline.join.requested.v1` 必须检查目标已存在，并匹配 `expectedStateRevision` 与 `expectedRegistryRevision`。加入只改变 actor 的时间线上下文，不代表地图、物品、资产或演化数值被复制。
- `timeline.registry.requested.v1` 的 `afterRevision` 表示消费端已经完整应用的 registry revision。结果必须返回当前 `registryRevision`、完整时间线目录和其后的创建事件；历史已被裁剪时必须设置 `truncated: true`，消费端应以完整目录重建本地视图。
- 空增量在 JSON 中应该表示为 `[]`；为兼容 Luanti 空表编码，`timeline.registry.snapshot.v1` 也接受 `events: null`，接收方必须将其规范化为空数组。

`timeline.created.v1` 是 v0.3 浏览器单 actor 分支语义，保留用于读取旧幂等响应。世界注册表创建必须使用 `timeline.created.v2`，不得改变 v1 已发布字段的含义。

`actorId` 必须是不透明平台身份，不应是玩家名、邮箱或登录名。权威平台持有显式的本地账户 ↔ actor 映射，并在处理命令时验证消息来源有权操作该 actor。ESIP 0.1 不定义全局账户系统；跨平台身份合并必须由额外的可信身份服务和用户授权完成，不能根据字符串相等推断。

跨游戏资产转移未来应使用 `prepare → commit → applied/rollback` 流程，并使用同一 correlation ID。ESIP 0.1 尚未标准化资产转移，适配器不得自行声称原子跨游戏交易。

## 5. 幂等、顺序和重试

发送方重试时必须复用原 `source + id` 和完全相同的消息内容。接收方必须满足：

- 相同 ID、相同内容只处理一次，并返回重复确认；
- 相同 ID、不同内容视为冲突并拒绝；
- 对要求有序的源，旧 `sequence` 或相同序列的新 ID 视为重放；
- 不能依赖“恰好一次”网络投递，业务处理本身必须幂等。

参考 `MemoryRouter` 只在内存中保存路由级去重和序列状态，进程重启后不会恢复。浏览器和 Luanti 权威适配器会另外持久化业务命令指纹及响应，使传输重投不会重复改变状态。生产传输仍必须持久化消费位置与去重记录，或由所有下游权威服务保证同等幂等语义。

## 6. 能力协商

适配器连接后应发布 `io.evolution.capability.hello.v1`，声明：

- 适配器 ID 和平台；
- 支持的 ESIP 版本；
- 可以消费和产生的消息类型；
- 最大消息字节数。

声明能力不等于获得权限。路由器仍必须对来源、目标、消息种类和业务身份执行授权。适配器禁止产生未声明类型。`target` 只是路由信息，不提供保密性；参考内存路由器中的订阅者必须被视为同一可信进程的一部分。

## 7. 安全与隐私

所有跨主机传输必须使用经过身份验证的加密连接。本机回环 HTTP 可以使用高熵 Bearer 令牌，但必须强制绑定 `127.0.0.1`/`::1`，不得降级为 LAN 或公网监听。协议边界必须默认拒绝 `command` 和 `query`，逐项配置许可。

实现必须限制消息大小、速率、处理时间和重试次数；检查过期时间、序列和 schema；使用不透明 `actorId`；在日志中删除令牌和个人信息。

ESIP `data` 禁止携带：

- 任意 Lua、JavaScript、shell 或可执行代码；
- API 密钥、会话令牌、密码和私钥；
- `auth.sqlite`、`players.sqlite` 或其他原始玩家数据库；
- 未经授权的聊天、定位或真实身份信息。

接入 openVirFactory 时，ESIP 应运行在 Node 侧车；Luanti 只连接本机受认证桥，并继续使用结构化命令和节点前缀白名单。

## 8. 版本与扩展

消息类型末尾的 `v1` 是 schema 主版本。ESIP 0.1 对 payload 使用严格字段白名单；新增或删除字段、改变含义或收紧既有合法值都必须发布新的消息主版本。只修改说明文字和不影响验证结果的 schema 注解可以保留主版本。

`esipversion` 描述核心外壳与语义版本。0.x 版本允许改进，但必须在 CHANGELOG 和迁移说明中记录。未知消息类型默认拒绝；其他游戏使用自己的反向域名命名空间，并向路由器和适配器同时注册 kind、dataschema 和运行时验证器。

## 9. 传输绑定

ESIP 与传输无关。仓库提供内存路由器用于验证语义，并提供带令牌的回环 HTTP sidecar 连接真实 Luanti。后续可以实现 WebSocket、NATS JetStream 或其他适配器，但传输层不得修改 `source + id`、事件内容或权威归属。
