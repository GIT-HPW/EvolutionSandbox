# 跨游戏通信接入指南

ESIP 的首要目标是隔离游戏平台差异。推荐把每个游戏的原生 API 留在本地适配器中，再让适配器之间交换统一消息。

```text
游戏进程 ←本机受控接口→ ESIP Adapter ←可替换传输→ Router/Broker
                                                    ↕
游戏进程 ←本机受控接口→ ESIP Adapter ←可替换传输→
```

## 当前可运行闭环

运行 `npm run demo:interop`。参考流程如下：

1. 网页控制端和 Luanti 规则端分别发布能力声明。
2. 网页端向 Luanti 端发送带目标、玩家上下文和期望版本的行为命令。
3. Luanti 规则端作为权威端执行既有确定性规则。
4. 执行成功后返回 `action.applied`；维度变化时额外产生 `realm.transitioned`。
5. 网页端发起状态查询，并收到带 revision 的状态快照。

这个演示不开放网络端口，适合验证 schema、权限、重复消息和规则映射。

## Luanti / Mineclonia

当前 `evolution_core` 不直接连接消息代理。生产接入建议复用 openVirFactory 的 Node Agent 作为本机侧车：

- Luanti 模组只访问 `127.0.0.1` 上经过令牌认证的结构化桥；
- Agent 将受控游戏状态映射为 ESIP；
- 远端消息先在 Agent 验证 ESIP、权限和白名单，再进入 Luanti；
- 禁止加入通用 `run_lua` 或把消息内容传给 `load`/`dofile`。

## Unity、Unreal 或其他游戏

新适配器至少实现：稳定 `source`、持久 sequence、能力声明、输入 schema 验证、授权、幂等存储、原生状态到 ESIP 的映射，以及断线重连策略。

不要尝试统一所有引擎对象。只交换业务事实，例如领域跃迁、时间线建立或已确认的玩家转移；渲染组件、物理帧和引擎内部实体仍由各自游戏管理。自定义消息使用 `com.example.game.event.v1` 风格命名空间，并注册自己的 schema 和运行时验证器。

## 从内存路由器升级

满足以下条件后才引入持久消息系统：

- 至少两个真实游戏平台已完成端到端往返；
- 已确定需要断线追赶或事件重放；
- 已定义消息保留期、访问控制和删除策略；
- 已压测消息大小、速率、重复和乱序；
- 已确定每类状态的唯一权威端。

升级传输时保留 ESIP JSON 语义，不让 broker 的主题、分区或交付特性泄漏进内容规则。
